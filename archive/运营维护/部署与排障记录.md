## 二、部署流程

### 2.1 首次部署

```bash
# 1. 克隆项目
cd /opt
sudo git clone https://github.com/1y4w1s/ai-industry-digest.git
sudo chown -R ubuntu:ubuntu ai-industry-digest

# 2. 配置环境变量
cd /opt/ai-industry-digest
nano .env
# 填入以下内容（从 .env.example 复制）:
# SUPABASE_URL=https://your-project.supabase.co
# SUPABASE_KEY=your-supabase-anon-key
# DEEPSEEK_API_KEY=your-deepseek-key
# DATABASE_URL=postgresql://...

# 3. 安装后端依赖
pip install -r requirements.txt

# 4. 构建前端
cd frontend
npm install && npm run build
```

### 2.2 启动服务

```bash
# 1. 启动后端
cd /opt/ai-industry-digest
source .env
nohup /home/ubuntu/.local/bin/uvicorn api.main:app --host 0.0.0.0 --port 8000 > backend.log 2>&1 &

# 2. 启动 Nginx
sudo systemctl start nginx

# 3. 验证
curl http://localhost:8000/api/sources       # 后端
curl http://localhost:8080/api/sources        # Nginx 代理
```

### 2.3 更新部署

```bash
cd /opt/ai-industry-digest
git stash                        # 暂存本地修改
git pull origin master           # 拉取最新代码
find . -name "__pycache__" -type d -exec rm -rf {} + 2>/dev/null  # 清除 Python 缓存
cd frontend && npm run build     # 重新构建前端
cd ..
pkill -f uvicorn                 # 停止旧服务
source .env
/home/ubuntu/.local/bin/uvicorn api.main:app --host 0.0.0.0 --port 8000 > backend.log 2>&1 &  # 启动新服务
sudo systemctl restart nginx     # 重启 Nginx
```

## 三、Nginx 配置

```
文件位置: /etc/nginx/sites-available/default
```

```nginx
server {
    listen 8080 default_server;
    listen [::]:8080 default_server;

    root /opt/ai-industry-digest/frontend/dist;
    index index.html;

    server_name _;
    include /etc/nginx/mime.types;

    # 前端路由支持
    location / {
        try_files $uri $uri/ /index.html;
    }

    # API 反向代理
    location /api/ {
        proxy_pass http://localhost:8000/api/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 静态文件缓存
    location ~* \.(js|css|png|jpg|svg|ico)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

### Nginx 配置要点

| 配置项 | 正确值 | 常见错误值 |
|--------|--------|------------|
| root | `/opt/ai-industry-digest/frontend/dist` | `/var/www/html` |
| proxy_pass | `http://localhost:8000/api/` | `http://localhost:3000` |
| listen | `8080` | `80` |
| mime.types | 必须 include | 遗漏导致 JS 加载为 text/html |

## 四、常见错误与修复记录

### 4.1 错误一：前端 JS 模块 MIME 类型错误

**现象**：浏览器控制台显示 `Failed to load module script: Expected a JavaScript-or-Wasm module script but the server responded with a MIME type of "text/html"`

**原因**：
1. Nginx 配置中 `root` 指向了错误的目录
2. JS 文件不存在（构建产物 hash 名称改变）
3. Nginx 配置中缺少 `include /etc/nginx/mime.types;`

**修复**：
- 确保 `root` 指向 `/opt/ai-industry-digest/frontend/dist`
- 重新运行 `npm run build`
- 确保 Nginx 配置包含 `include /etc/nginx/mime.types;`

---

### 4.2 错误二：API 500 — UUID 字段类型不匹配（重点）

**现象**：所有 `/api/auth/*` 接口返回 500，后端日志显示：
```
postgrest.exceptions.APIError: {'code': '22P02', 'message': 'invalid input syntax for type uuid: "demo-user"'}
```

**根因**：前端未登录时发送 `demo-user` 作为用户标识，数据库 `user_id` 字段为 UUID 类型，无法接收非 UUID 字符串。

**问题链路**：
```
前端请求 → Authorization: Bearer demo-user → 后端接收 → 直接传入 Supabase 查询
→ PostgreSQL 报错 22P02 → 后端返回 500 → 前端显示 500
```

#### 修复文件

**后端** (`api/routes/auth.py`)：

```python
# 在文件顶部添加常量
DEMO_USER_ID = "demo-user"
DEMO_USER_UUID = "00000000-0000-0000-0000-000000000001"

def get_user_id(authorization: str = Header(None)) -> str:
    if not authorization:
        raise HTTPException(status_code=401, detail="未登录")

    token = authorization.replace("Bearer ", "")

    # Demo 用户特殊处理：转换为有效的 UUID
    if token == DEMO_USER_ID:
        return DEMO_USER_UUID

    # 尝试解析 JWT token
    try:
        decoded = jwt.decode(token, options={"verify_signature": False})
        user_id = decoded.get("sub")
        if user_id:
            return user_id
    except Exception:
        pass

    return token
```

**前端** (`frontend/src/api/client.js`)：

```javascript
const DEMO_TOKEN = 'demo-user';

function getToken() {
  try { return localStorage.getItem(TOKEN_KEY); } catch { return null; }
}

async function request(path, options = {}) {
  const token = getToken() || DEMO_TOKEN;  // 未登录则用 demo-token
  // ...
  headers['Authorization'] = `Bearer ${token}`;
  // ...
}
```

**修复后链路**：
```
前端请求 → Authorization: Bearer demo-user → 后端 get_user_id() 检测到 demo-user
→ 返回 00000000-0000-0000-0000-000000000001 → Supabase 查询成功 → 返回 200
```

### 4.3 错误三：supabase.auth.session is not a function

**现象**：前端报错 `Uncaught TypeError: supabase.auth.session is not a function`

**原因**：Supabase JS SDK v2 废弃了 `session()` 方法，改用 `getSession()`

**修复**：在 `AuthContext.jsx` 中将 `supabase.auth.session()` 替换为 `supabase.auth.getSession()`

---

### 4.4 错误四：前端请求后端 Connection Refused

**现象**：请求 `http://localhost:8000/api/*` 提示 `ERR_CONNECTION_REFUSED`

**原因**：开发时前端使用了绝对路径 `http://localhost:8000/api`，生产环境应使用相对路径通过 Nginx 反向代理访问。

**修复**：将 `.env` 中的 `VITE_API_URL` 从 `http://localhost:8000/api` 改为 `/api`

---

### 4.5 错误五：Supabase 登录 400 Bad Request

**现象**：登录时报 `POST /auth/v1/token?grant_type=password 400 (Bad Request)`

**原因**：Supabase 默认开启了邮箱验证，开发环境中无法接收验证邮件。

**解决**：在 Supabase 控制台 → Authentication → Settings → 关闭 "Confirm email"

### 4.6 错误六：聊天接口 500 — article_list 格式为分组 dict 非 list

**现象**：点击 AI 对话发送消息 → 控制台 500，后端日志：
```
AttributeError: 'str' object has no attribute 'get'
```
或
```
TypeError: unhashable type: 'slice'
```

**两次修复合集**：

| 尝试 | 报错 | 原因 |
|------|------|------|
| 第一次 | `TypeError: unhashable type: 'slice'` | `get_report_by_date()` 返回的 `articles` 是 Supabase 特殊类型，不支持 `[:10]` 切片 |
| 第二次 | `AttributeError: 'str' object has no attribute 'get'` | `articles` 实际上是分组 dict `{"high": [...], "medium": [...], "low": [...]}`，遍历 dict 得到的是字符串 key |

**根因**：`get_report_by_date()` 在 `database.py` 中将文章按重要性分组后以 dict 形式存入 `report["articles"]`。遍历 dict 得到的是 key 字符串（"high"/"medium"/"low"），而非文章对象。

**修复方案**（`api/routes/chat.py`）：

```python
# 获取分组后的文章 dict
article_groups = report_detail.get("articles", {})
if isinstance(article_groups, dict):
    # 按重要性顺序扁平化为单层列表
    flat = []
    for priority in ("high", "medium", "low"):
        flat.extend(article_groups.get(priority, []))
    article_items = flat[:10]  # 现在可以正常切片了
```

**修复后链路**：
 ```
 report["articles"] = {"high": [...], "medium": [...], "low": [...]}
   → 遍历 high/medium/low 顺序展开
   → flat = [(high文章), (medium文章), (low文章)]
   → 取前 10 篇 → 格式化为 Prompt → 注入对话上下文
 ```

### 4.7 错误七：AI 回复中 **加粗** 等 markdown 标记显示为原始文本

**现象**：AI 回复中包含 `**内容**` 等 markdown 标记时，前端直接显示为原始文本 `**内容**`，用户在手机上看会感觉不适或困惑。

**根因**：所有 AI 回复使用 `{msg.content}` 纯文本插值渲染，React 自动对 HTML 进行转义，markdown 符号未经解析直接输出。

**修复方案**：

1. 新建轻量级 markdown → HTML 渲染函数（`frontend/src/utils/markdown.js`）：

```javascript
export function renderMd(text) {
  if (!text) return '';
  return text
    .replace(/&/g, '&amp;').replace(/</g, '&lt;').replace(/>/g, '&gt;')  // XSS 防护
    .replace(/\*\*(.+?)\*\*/g, '<strong>$1</strong>')                     // **加粗**
    .replace(/\*(.+?)\*/g, '<em>$1</em>')                                 // *斜体*
    .replace(/`(.+?)`/g, '<code style="...">$1</code>')                  // `代码`
    .replace(/\n/g, '<br/>');                                             // 换行
}
```

2. 将 3 个对话组件的纯文本渲染改为 HTML 渲染：

| 组件 | 文件 | 改动 |
|------|------|------|
| 文章阅读器右侧聊天 | `ArticleReader.jsx` | `{msg.content}` → `<span dangerouslySetInnerHTML={{ __html: renderMd(msg.content) }} />` |
| AI 推荐面板 | `AIRecommendPanel.jsx` | 同上 |
| 首页浮动 AI 气泡 | `AIChatBubble.jsx` | 同上 |

**安全考虑**：`renderMd()` 先对 `&` `<` `>` 做 HTML 转义，再替换 markdown 标记，防止 XSS 注入。仅渲染 AI 回复（assistant 角色），用户输入（user 角色）仍使用纯文本。

**修复后效果**：
```
AI 回复 "**字节AI** 2026年四大战略" → 显示为 "字节AI 2026年四大战略"
```

### 4.8 错误八：量子位等 RSS 来源的文章只显示标题/摘要，无完整正文

**现象**：如量子位（qbitai.com）的文章在平台内只显示第一句话（如"聚焦高校科研算力"），点击阅读后正文只有寥寥数语。

**根因**：量子位的 RSS Feed 只返回文章摘要（summary/description），而非完整正文。采集器直接使用 RSS 内容，导致 `raw_content` 只有一句话。

**修复方案**（`collector/rss_collector.py`）：

1. 新增 `_fetch_full_content()` 方法——当 RSS 内容不足 100 字符时，从文章 URL 爬取完整正文
2. 使用 requests + BeautifulSoup 抓取，按优先级匹配 10 种常见正文 CSS 选择器：
   - `article`、`.entry-content`、`.post-content`、`.article-content`、`.rich_media_content` 等
3. 自动移除正文中的 `script`、`style`、`nav`、`footer`、`广告`等无用元素
4. 降级方案：取 `<body>` 内所有文本（兜底）

**爬取条件**：
```python
MIN_CONTENT_LENGTH = 100  # RSS 内容 < 100 字符时触发爬取
```

**调用链路**：
```
RSS _parse_entry() / _parse_raw_rss()
  → 判断 len(content) < 100
  → _fetch_full_content(url)
  → requests.get() + BeautifulSoup
  → 匹配 10 种选择器 → 返回正文
  → 替换 RSS 摘要
```

**效果**：
```
之前：raw_content = "聚焦高校科研算力"
之后：raw_content = "5月29日至31日，2026年中国图象图形大会（CCIG 2026）在广州举办。本次大会由中国图象图形学会主办...（完整正文）"
```

**影响来源**：量子位、机器之心、36氪等所有 RSS 来源（arXiv 不受影响，走 API 直接返回完整摘要）

## 七、重要注意事项

1. **构建产物命名**：每次 `npm run build` 生成的 JS/CSS 文件名 hash 不同，这是正常现象。旧版本页面可能因缓存加载旧 JS 文件，需要 `Ctrl+Shift+R` 强制刷新。

2. **Python 模块缓存**：修改 Python 代码后，清除 `__pycache__` 目录确保新代码生效：
   ```bash
   find /opt/ai-industry-digest -name "__pycache__" -type d -exec rm -rf {} +
   ```

3. **Demo 用户**：系统内置了一个 demo 用户（UUID: `00000000-0000-0000-0000-000000000001`），用于未登录时的浏览体验。所有未登录用户的浏览历史和收藏共享此 demo 账户。

4. **API 认证流程**：
   - 已登录用户：使用 Supabase JWT token
   - 未登录用户：使用 `demo-user` token（后端自动转换为 UUID）
   - 认证方式：`Authorization: Bearer <token>`
