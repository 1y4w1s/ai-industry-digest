## 二、部署流程

### 2.1 首次部署

```bash
# 1. 克隆项目
cd /opt
sudo git clone https://github.com/1y4w1s/ai-industry-digest.git
sudo chown -R ubuntu:ubuntu ai-industry-digest

# 2. 配置环境变量
cd /opt/ai-industry-digest
nano .env
# 填入以下内容（从 .env.example 复制）:
# SUPABASE_URL=https://your-project.supabase.co
# SUPABASE_KEY=your-supabase-anon-key
# DEEPSEEK_API_KEY=your-deepseek-key
# DATABASE_URL=postgresql://...

# 3. 安装后端依赖
pip install -r requirements.txt

# 4. 构建前端
cd frontend
npm install && npm run build
```

### 2.2 启动服务

```bash
# 1. 启动后端
cd /opt/ai-industry-digest
source .env
nohup /home/ubuntu/.local/bin/uvicorn api.main:app --host 0.0.0.0 --port 8000 > backend.log 2>&1 &

# 2. 启动 Nginx
sudo systemctl start nginx

# 3. 验证
curl http://localhost:8000/api/sources       # 后端
curl http://localhost:8080/api/sources        # Nginx 代理
```

### 2.3 更新部署

```bash
cd /opt/ai-industry-digest
git stash                        # 暂存本地修改
git pull origin master           # 拉取最新代码
find . -name "__pycache__" -type d -exec rm -rf {} + 2>/dev/null  # 清除 Python 缓存
cd frontend && npm run build     # 重新构建前端
cd ..
pkill -f uvicorn                 # 停止旧服务
source .env
/home/ubuntu/.local/bin/uvicorn api.main:app --host 0.0.0.0 --port 8000 > backend.log 2>&1 &  # 启动新服务
sudo systemctl restart nginx     # 重启 Nginx
```

## 三、Nginx 配置

```
文件位置: /etc/nginx/sites-available/default
```

```nginx
server {
    listen 8080 default_server;
    listen [::]:8080 default_server;

    root /opt/ai-industry-digest/frontend/dist;
    index index.html;

    server_name _;
    include /etc/nginx/mime.types;

    # 前端路由支持（index.html 禁用缓存，避免部署后加载旧 hash 文件）
    location / {
        try_files $uri $uri/ /index.html;
    }
    location = /index.html {
        add_header Cache-Control "no-cache, no-store, must-revalidate";
    }

    # API 反向代理
    location /api/ {
        proxy_pass http://localhost:8000/api/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        # 大文件预览超时保护（PDF/DOCX 处理可能超过默认 60s）
        proxy_read_timeout 300;
        proxy_send_timeout 300;
    }

    # 静态文件缓存
    location ~* \.(js|css|png|jpg|svg|ico)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
```

### Nginx 配置要点

| 配置项 | 正确值 | 常见错误值 |
|--------|--------|------------|
| root | `/opt/ai-industry-digest/frontend/dist` | `/var/www/html` |
| proxy_pass | `http://localhost:8000/api/` | `http://localhost:3000` |
| listen | `8080` | `80` |
| mime.types | 必须 include | 遗漏导致 JS 加载为 text/html |

## 四、常见错误与修复记录

### 4.1 错误一：前端 JS 模块 MIME 类型错误

**现象**：浏览器控制台显示 `Failed to load module script: Expected a JavaScript-or-Wasm module script but the server responded with a MIME type of "text/html"`

**原因**：
1. Nginx 配置中 `root` 指向了错误的目录
2. JS 文件不存在（构建产物 hash 名称改变）
3. Nginx 配置中缺少 `include /etc/nginx/mime.types;`

**修复**：
- 确保 `root` 指向 `/opt/ai-industry-digest/frontend/dist`
- 重新运行 `npm run build`
- 确保 Nginx 配置包含 `include /etc/nginx/mime.types;`

---

### 4.2 错误二：API 500 — UUID 字段类型不匹配（重点）

**现象**：所有 `/api/auth/*` 接口返回 500，后端日志显示：
```
postgrest.exceptions.APIError: {'code': '22P02', 'message': 'invalid input syntax for type uuid: "demo-user"'}
```

**根因**：前端未登录时发送 `demo-user` 作为用户标识，数据库 `user_id` 字段为 UUID 类型，无法接收非 UUID 字符串。

**问题链路**：
```
前端请求 → Authorization: Bearer demo-user → 后端接收 → 直接传入 Supabase 查询
→ PostgreSQL 报错 22P02 → 后端返回 500 → 前端显示 500
```

**修复文件**：`api/routes/auth.py` — 添加 `DEMO_USER_UUID` 常量转换

### 4.3 错误三：supabase.auth.session is not a function

**现象**：前端报错 `Uncaught TypeError: supabase.auth.session is not a function`

**原因**：Supabase JS SDK v2 废弃了 `session()` 方法，改用 `getSession()`

**修复**：在 `AuthContext.jsx` 中将 `supabase.auth.session()` 替换为 `supabase.auth.getSession()`

### 4.4 错误四：前端请求后端 Connection Refused

**现象**：请求 `http://localhost:8000/api/*` 提示 `ERR_CONNECTION_REFUSED`

**原因**：开发时前端使用了绝对路径 `http://localhost:8000/api`，生产环境应使用相对路径通过 Nginx 反向代理访问。

**修复**：将 `.env` 中的 `VITE_API_URL` 从 `http://localhost:8000/api` 改为 `/api`

### 4.5 错误五：Supabase 登录 400 Bad Request

**现象**：登录时报 `POST /auth/v1/token?grant_type=password 400 (Bad Request)`

**原因**：Supabase 默认开启了邮箱验证，开发环境中无法接收验证邮件。

**解决**：在 Supabase 控制台 → Authentication → Settings → 关闭 "Confirm email"

### 4.6 错误六：JWT Token 验证失败（2026-06-07）

**现象**：
- 所有 `/api/auth/*` 接口返回 401 Unauthorized
- 后端日志显示：`PyJWKClientConnectionError: Fail to fetch data from the url`
- 用户无法正常认证，浏览历史和收藏功能失效

**根因**：
1. 后端使用手动方式获取 JWKS（JSON Web Key Set）验证 Token
2. Supabase 的 JWKS 端点从 `/.well-known/jwks.json` 变更为 `/auth/v1/jwks`
3. 新端点需要 API Key 认证才能访问

**问题链路**：
```
前端请求 → Authorization: Bearer <token> → 后端 JWT 验证 → JWKS 获取失败 → 返回 401
```

**修复文件**：`api/services/jwt_verify.py`

**修复方案**：
- 移除 PyJWKClient 依赖和手动 JWT 解码
- 使用 Supabase 官方 SDK 的 `auth.get_user(token)` 方法验证 Token
- 避免手动管理 JWKS 端点和密钥轮换

**关键代码变更**：
```python
# 修复前（问题代码）
jwks_client = PyJWKClient(JWKS_URL)
signing_key = jwks_client.get_signing_key_from_jwt(token)
decoded = jwt.decode(token, signing_key.key, algorithms=["HS256", "RS256"])

# 修复后
supabase = create_client(SUPABASE_URL, SUPABASE_KEY)
response = supabase.auth.get_user(token)
if response.user and response.user.id:
    return response.user.id
```

### 4.7 错误七：数据库列不存在导致浏览历史失败（2026-06-07）

**现象**：
- 浏览历史接口返回 200 OK，但数据未写入数据库
- 后端日志显示：`[DB] 添加浏览历史失败: {'code': '42703', 'message': 'column reading_history.read_percent does not exist'}`
- 用户阅读文章后，浏览历史不会出现新记录

**根因**：
1. 代码中引用了 `read_percent` 和 `duration_sec` 列
2. 数据库表 `reading_history` 中未创建这些列
3. SELECT 查询和 INSERT/UPDATE 操作都引用了不存在的列

**问题链路**：
```
用户阅读文章 → 前端调用 POST /api/auth/history → 后端执行 SELECT 查询 → 
查询 read_percent 列失败 → 捕获异常返回 False → 数据未写入
```

**修复文件**：`api/models/database.py`

**修复方案**：
1. SELECT 查询只选择存在的列（`id`），避免引用不存在的 `read_percent`
2. UPDATE 和 INSERT 操作使用 try-except 进行降级处理
3. 列不存在时，只使用基本字段进行操作

**关键代码变更**：
```python
# 修复前（问题代码）
existing = self.client.table("reading_history") \
    .select("id, read_percent") \  # 问题：查询不存在的列
    ...

# 修复后
existing = self.client.table("reading_history") \
    .select("id") \  # 只查询存在的列
    ...

# 插入数据时的降级处理
try:
    data = {"user_id": user_id, "article_id": article_id, "read_percent": read_percent}
    self.client.table("reading_history").insert(data).execute()
except Exception:
    # 降级：列不存在时只插入基本数据
    data = {"user_id": user_id, "article_id": article_id}
    self.client.table("reading_history").insert(data).execute()
```

### 4.8 优化：Nginx Gzip 压缩优化（2026-06-07）

**现象**：
- 移动端首次加载极慢，主入口 JS 文件下载耗时过长
- 主入口文件 `index-T45QGQ17.js` 大小约 929KB
- 在 4G 网络下下载时间约 750ms，3G 网络下约 2.5s

**性能分析**：
| 文件 | 原始大小 | Gzip 后 | 压缩率 |
|------|----------|---------|--------|
| 主入口 JS | 929KB | 276KB | -70.3% |
| Supabase SDK JS | 201KB | 60KB | -70.1% |
| 主 CSS | 25KB | 7KB | -72.0% |

**根因**：Nginx 未启用 Gzip 压缩，导致静态资源传输体积过大，移动端首次加载缓慢。

**Nginx 配置位置**：
- 主配置：`/etc/nginx/nginx.conf`
- 站点配置：`/etc/nginx/sites-available/signal`

**修复方案**：

1. **配置 Nginx Gzip 压缩**：
```nginx
gzip on;
gzip_vary on;
gzip_proxied any;
gzip_comp_level 6;
gzip_buffers 16 8k;
gzip_http_version 1.1;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript application/x-javascript application/font-woff2 image/svg+xml;
```

2. **配置说明**：
| 配置项 | 值 | 说明 |
|--------|-----|------|
| `gzip on` | 启用 | 开启 Gzip 压缩 |
| `gzip_vary on` | 启用 | 添加 Vary 头，支持 CDN 缓存不同编码版本 |
| `gzip_proxied any` | any | 压缩所有代理请求 |
| `gzip_comp_level` | 6 | 压缩级别（1-9，6 为平衡） |
| `gzip_buffers` | 16 8k | 压缩缓冲区配置 |
| `gzip_http_version` | 1.1 | 支持 HTTP/1.1 |
| `gzip_types` | ... | 需要压缩的 MIME 类型 |

3. **验证压缩是否生效**：
```bash
# 测试脚本：scripts/test_gzip.py
python3 scripts/test_gzip.py
```

**执行步骤**：
1. 检查当前配置：`curl -I -H "Accept-Encoding: gzip" http://43.139.133.245:8080/assets/index-T45QGQ17.js`
2. 更新 Nginx 配置（主配置和站点配置）
3. 测试配置：`sudo nginx -t`
4. 重启 Nginx：`sudo systemctl restart nginx`
5. 验证压缩效果

**性能提升**：
- 移动端 4G 网络：下载时间从 ~750ms 降至 ~220ms（提升 70.7%）
- 移动端 3G 网络：下载时间从 ~2.5s 降至 ~0.7s（提升 72%）
- 单次加载节省带宽：约 653KB（主入口文件）

**相关脚本**：
- `scripts/fix_nginx_gzip_v2.py`：自动配置 Gzip
- `scripts/test_gzip.py`：验证压缩效果

## 七、重要注意事项

1. **构建产物命名**：每次 `npm run build` 生成的 JS/CSS 文件名 hash 不同，这是正常现象。旧版本页面可能因缓存加载旧 JS 文件，需要 `Ctrl+Shift+R` 强制刷新。

2. **Python 模块缓存**：修改 Python 代码后，清除 `__pycache__` 目录确保新代码生效：
   ```bash
   find /opt/ai-industry-digest -name "__pycache__" -type d -exec rm -rf {} +
   ```

3. **Demo 用户**：系统内置了一个 demo 用户（UUID: `00000000-0000-0000-0000-000000000001`），用于未登录时的浏览体验。所有未登录用户的浏览历史和收藏共享此 demo 账户。

4. **API 认证流程**：
   - 已登录用户：使用 Supabase JWT token
   - 未登录用户：使用 `demo-user` token（后端自动转换为 UUID）
   - 认证方式：`Authorization: Bearer <token>`
