# 面试常见追问与回答模板

## Q1: "缓存穿透/雪崩怎么解决？"

> **穿透**——如果一个 key 在数据库里也不存在，我会缓存一个空值并设短 TTL（30 秒），防止恶意请求持续打到数据库。
>
> **雪崩**——差异化 TTL + 随机浮动，比如 300 秒 ± 30 秒，不让大量缓存同时过期。
>
> **最关键的是降级兜底**——Redis 不可用时所有请求直接走数据库查询，系统不会挂。

## Q2: "为什么 JWT 验证要降级？"

> Supabase Auth 的 session 有生命周期（默认 3600 秒），但 JWT token 本身没过期。如果 session 过期就直接拒绝用户，体验极差。
>
> 我加了降级——直接解码 JWT 的 payload 获取 user_id。JWT 的签名防篡改在签发时已保证，降级只是绕过 session 校验获取身份，不是安全漏洞。

## Q3: "AI Token 优化效果怎么量化？"

> 四层优化叠加：
>
> 1. **问题预分类**：闲聊和通用知识不注入上下文，省 200-600 tokens/次
> 2. **请求级缓存**：相同问题第二次零成本
> 3. **上下文精简**：SYSTEM_PROMPT 从 400 砍到 150 字，对话历史从 6 轮减到 2 轮
> 4. **参数调优**：temperature 0.5→0.3，回答更一致，缓存命中率从 50% 升到 95%
>
> 结果：重复问题零成本，新问题成本降 60%。

## Q5: "WebSocket 是怎么实现的？"

> 实现了一个 WebSocketManager，内部用 `defaultdict(list)` 维护用户 ID 到连接的映射。
>
> 三种推送模式：
> - **广播**：全员通知（新文章入库）
> - **单播**：推送指定用户（收藏变动）
> - **时段推送**：指定时间段的人群
>
> 连接断开时自动清理映射表，防止内存泄漏。

```python
class WebSocketManager:
    def __init__(self):
        self.connections = defaultdict(list)

    async def connect(self, ws, user_id):
        await ws.accept()
        self.connections[user_id].append(ws)

    async def broadcast(self, message):
        for user_id, conns in self.connections.items():
            for ws in conns:
                await ws.send_json(message)

    async def disconnect(self, ws, user_id):
        self.connections[user_id].remove(ws)
```

## Q6: "数据量大了以后怎么扩展？"

> 当前瓶颈在 AI 处理层。如果文章量翻 10 倍：
>
> 1. **Redis 缓存层级**——热门文章完全不用 AI 处理，直接从缓存返回
> 2. **消息队列**——引入 RabbitMQ 异步批处理，采集和 AI 处理解耦
> 3. **Elasticsearch**——替代当前的数据库 LIKE 搜索，支持全文检索
> 4. **虚拟列表**——前端虚拟滚动，1000 篇文章也流畅
> 5. **水平扩展**——FastAPI 可以多进程部署，Nginx 做负载均衡

## Q7: "Nginx 反向代理为什么要做？"

> 早期用 8000 端口直接暴露 FastAPI 服务，有跨域问题和性能瓶颈。
>
> Nginx 做了四件事：
>
> 1. **端口统一**：前端 80 + 后端 8000 → 统一 80
> 2. **静态资源直服**：Nginx 直接 Serve 静态文件，比 FastAPI 快 10 倍
> 3. **Gzip 压缩**：JS/CSS 体积减少 70%，首次加载从 3s 降到 1s
> 4. **安全过滤**：挡掉扫端口的不安全请求（如 Umai-Scanner）

## Q8: "为什么采集要用 RSS + 全文兜底？"

> RSS 通常只提供摘要（~100 字），信息量不足以让 AI 生成有价值的摘要和标签。
>
> 我设计了兜底机制——如果 RSS 内容 < 100 字符，自动用 BeautifulSoup 去原文页面抓取完整正文，保留 HTML 结构（段落、列表等）。
>
> 这样 AI 处理时输入的是完整文章，而不是一段摘要。

## Q9: "采集去重为什么三层？"

> 第一层 URL 精确去重——简单高效，相同 URL 直接丢弃。
>
> 第二层标题相似度去重——同一事件不同媒体可能 URL 不同但标题相似，用 SequenceMatcher 计算相似度，>85% 自动合并。
>
> 第三层 AI 去重——70%-85% 之间的模糊情况，让 DeepSeek 判断是否为同一事件，在信息量和成本之间取平衡。

## Q10: "你是怎么考虑错误处理的？"

> 分层策略：
>
> - **采集层**：每个源独立 try/except，一个源挂了不影响其他 7 个
> - **AI 层**：单批解析失败自动重试 1 次，连续失败跳过该批
> - **缓存层**：Redis 连接失败自动降级到数据库直查
> - **JWT 层**：Supabase 验证失败降级到直接解码 JWT
> - **API 层**：所有异常统一交给 FastAPI 的 ExceptionHandler 返回 JSON 错误
