## 一、整体架构

### Q1：项目整体架构是怎么样的？
```
采集层（8个源）→ 去重层（3层）→ AI处理层（批处理）→ 日报生成 → 入库
```
前端 React + Vite → Nginx 反向代理 → FastAPI 后端 → Supabase 存储 + Redis 缓存

### Q2：你的角色是什么？负责哪些模块？
> **"独立开发，全栈。后端主攻采集管道、AI 批处理、缓存、JWT 认证；前端负责首页、日报展示、AI 对话、知识图谱。"**

### Q3：技术栈选型的理由？
| 技术 | 为什么选 |
|------|---------|
| FastAPI | 异步支持好，自动生成 OpenAPI 文档，性能足够 |
| DeepSeek | 性价比高（对比 GPT-4o 便宜 10 倍），国产稳定 |
| Supabase | BaaS 免运维，自带 Auth + PostgreSQL |
| Redis | 缓存降级设计，AI 对话去重 |
| React + Vite | 轻量，HMR 快 |

## 二、采集层

### Q4：采集层怎么设计的？
> **策略模式 + 配置驱动。** `BaseCollector` 抽象基类定义 `collect()` 接口，RSS/arXiv/HF 各自实现。`sources.yaml` 配源，新增源只写类 + 改 yaml。

### Q5：RSS 内容太短怎么办？
> **全文抓取兜底。** RSS 摘要 < 100 字符时，自动用 BeautifulSoup 抓原文。优先级尝试 `article` → `.entry-content` → `.rich_media_content` → `main` 等选择器。

### Q6：arXiv 采集有什么限制？
> **限流 3 秒 1 次**（arXiv 要求），每次拉取 50 篇 cs.AI 类别最新提交。

### Q7：一个源挂了怎么办？
> **每个源独立 try/except。** ThreadPoolExecutor 并发采集，`as_completed` 拿到每个源的结果。一个源挂了只跳过它自己，不影响其他 7 个。采集完输出成功/失败统计。

### Q8：为什么并发采集用 4 个线程？
> **实测值。** 8 个源串行 40s → 4 线程并行 12s（快 70%）。再多线程也没用，瓶颈在 HTTP 请求的网络 IO 不是 CPU。

### Q9：采集到的重复数据怎么处理？（高频）
> **入库时用 URL 唯一键去重**（数据库层 `eq("url", article.url)` 检查），配合采集层的三层去重，双重保障。

## 三、去重层（最高频考点）

### Q10：三层去重具体怎么做的？
```
第一层：URL 精确去重 → set 判重，O(1)
第二层：标题相似度去重 → SequenceMatcher，≥85% 自动合并
第三层：AI 辅助去重 → 70%-85% 区间送 DeepSeek 判断
```

### Q11：85% 和 70% 阈值怎么定的？
> **数据驱动。** 标了 200 篇历史数据，发现 85% 以上误判率 < 5%，70% 以下送 AI 准确率只有 50%（不值得花 tokens）。15% 的边界文章用 AI 判断，准确率 80%+。

### Q12：SequenceMatcher 的原理了解吗？
> **基于最长公共子序列（LCS）算法**。`ratio = 2 * M / (len(a) + len(b))`，M 是 LCS 长度。时间复杂度 O(n²)，但标题一般 < 50 字，性能可接受。

### Q13：AI 去重失败怎么办？
> **保守策略**：两篇文章都保留，标记 `suspected_duplicate=True`。宁可多展示一篇，也不删可能有用的信息。

### Q14：合并重复文章时保留哪篇？
> **保留信息更完整的那篇**：`_merge_articles()` 把 source_refs 追加到目标文章，如果目标没正文就用来源的正文。

### Q15：为什么没做内容相似度去重？
> **性价比问题。** 标题已经能覆盖 90% 的重复场景。算全文相似度需要把正文读到内存，200 篇文章 O(n²) 对比，内存和 CPU 开销大，收益只有不到 5%。

## 四、AI 处理层

### Q16：AI 一次处理多少篇？为什么？
> **10 篇一批。** 实测对比：5 篇调用太频繁，20 篇输出格式不稳定（偶尔漏字段），10 篇刚好，格式稳定率 99%+。

### Q17：一次调用同时完成什么？
> **4 件事一次做完**：摘要生成 + 标签提取 + 重要性评分 + 文章分类。比分开调省 60%+ 成本。

### Q18：批处理的 token 消耗？
```
每批：4,500 tokens（系统提示 ~300 + 文章内容 ~2,200 + 输出 ~2,000）
每次运行（13-16 批）：~60,000-72,000 tokens
每日（3 次）：~216,000 tokens
```

### Q19：为什么不用单篇调用？
> **成本对比**：批处理 1 次 × 4,500 tokens vs 单篇 10 次 × 1,200 tokens = 省 63%。而且 10 次调用有 10 次网络延迟。

### Q20：AI 处理失败怎么兜底？
> **指数退避重试 3 次（4s, 8s）**。重试耗尽后填充默认值：
> - summary = `[AI处理失败] {title}`
> - tags = `["其他"]`
> - importance = `"low"`
> 
> 保证入库流程不中断，等下次管道运行重新处理。

### Q21：为什么 temperature 设 0.3？
> **输出一致性优先。** 0.3 保证 JSON 格式稳定，不会出现格式漂移整批报错。如果是创意场景才会调高。

### Q22：DeepSeek API 不稳定怎么办？
> - **超时**：120s 超时，超了自动重试
> - **限流 429**：等待 2^attempt 秒后重试
> - **密钥无效 401**：直接返回 None，不重试（省时间）
> - **重试 2 次仍失败**：返回 None，上层 `_process_with_retry` 接管兜底

### Q23：为什么 raw_content 只取前 800 字？
> **控制 token 成本。** 原文可能很长（3000+ 字），但 AI 摘要只需要关键信息。800 字足够生成质量合格的摘要，省了一半 token。

## 五、缓存层

### Q24：缓存怎么设计的？
> **CacheService 封装**，两种模式：
> 1. 数据库查询缓存：首页 5 分钟、日报 30 分钟
> 2. AI 对话缓存：通用知识 1 天、闲聊 1 小时、日报 1 小时、文章 30 分钟

### Q25：Redis 挂了怎么办？
> **自动降级**：构造方法 try Redis 连接失败 → `self.available = False` → 所有 get/set 直接 return。用户无感知，只是响应慢了 200ms。

### Q26：怎么防止缓存雪崩？
> 1. **TTL 随机浮动**：300s ± 30s
> 2. **降级兜底**：Redis 挂了直接查库
> 3. **熔断**：连续报错自动暂停缓存

### Q27：AI 对话缓存的命中率多少？
> **实测 95%+**。因为 temperature 0.3 输出一致性高，相同问题的回答几乎一样，MD5 作为 key 命中率很高。

### Q28：缓存 key 怎么设计的？
> `{prefix}:{md5(question)}`。前缀区分类型（general/chat/daily/article），MD5 保证等长且无特殊字符。

## 六、JWT 认证

### Q29：认证流程是怎么样的？
```
demo 用户 → 直接放行（浏览公开内容）
↓
JWT token → Supabase auth.get_user() 验证
  ↓ 成功 → 返回 user_id
  ↓ 失败 → 直接解码 JWT payload 获取 sub（fallback）
↓
全部失败 → 401
```

### Q30：为什么 JWT 要降级？
> Supabase Auth session 有生命周期（3600s），但 JWT token 本身没过期。session 过期就拒绝用户，体验极差。降级只从 payload 取 user_id，签名防篡改在签发时已保证。

### Q31：不验证签名直接解码 JWT 安全吗？
> **安全，但有前提。** 这只在 Supabase 验证失败时兜底。JWT 的签名是在签发时由 Supabase 做的，我们 payload 里只有 user_id，没有敏感信息。相当于绕过 session 校验获取身份，不是绕过签名验证。

### Q32：demo 用户的设计？
> 未登录用户用 `00000000-0000-0000-0000-000000000001` UUID，只能浏览公开的日报内容，不能收藏、不能对话。

## 七、AI 对话

### Q33：AI 对话怎么优化 token 消耗？（高频）
> **四层优化：**
> 1. **问题预分类**：闲聊/通用知识不注入上下文，省 200-600 tokens/次
> 2. **请求级缓存**：相同问题第二次零成本
> 3. **上下文精简**：SYSTEM_PROMPT 400→150 字，对话历史 6→2 轮
> 4. **参数调优**：temperature 0.5→0.3，缓存命中率 50%→95%
>
> 结果：**重复问题零成本，新问题成本降 60%。**

### Q34：问题预分类怎么实现的？
> 用关键词规则 + 分类 prompt。规则匹配不到才调分类 API。比如 "你好"、"在吗" 直接走闲聊，不问日报相关就不注入日报数据。

### Q35：对话历史怎么管理的？
> 只保留最近 2 轮（4 条消息），用 Redis 存。过期时间 1 小时。太长会撑爆 token 上限，太短没有上下文连贯性。

## 八、数据库

### Q36：用什么数据库？为什么选 Supabase？
> PostgreSQL（通过 Supabase）。**免运维 + Auth 内置 + 实时订阅功能**。个人项目不需要自己搭数据库。

### Q37：数据库表结构？
| 表名 | 用途 |
|------|------|
| articles | 文章主表，url 有唯一索引 |
| daily_reports | 生成的日报 |
| user_profiles | 用户信息 |
| bookmarks | 用户收藏 |
| reading_history | 浏览记录 |

### Q38：写入文章怎么去重？
> 先 `select("id").eq("url", article.url)` 检查，存在就 skip，不存在才 insert。**写入前 + 写入时双层去重**。

### Q39：分页怎么做的？
> `.range((page-1) * page_size, page * page_size - 1)`，配合 count 返回总数。

### Q40：数据库操作有重试吗？
> 有。`_execute_with_retry` 最大 3 次，指数退避 2s/4s/8s，每次重试重新创建客户端（应对连接断开）。

## 九、WebSocket

### Q41：WebSocket 用来做什么？
> **实时推送通知**。收藏变动、浏览历史更新时，推送给对应用户。

### Q42：怎么管理连接？
> `WebSocketManager` 用 `defaultdict(list)` 维护 `user_id → [ws1, ws2, ...]` 映射。单用户多端登录都能收到。

### Q43：断开连接怎么清理？
> `disconnect()` 从映射表删除对应连接。用户全部断开后自动清理空列表。

### Q44：支持哪些推送模式？
> - **广播**：全员通知（新文章入库）
> - **单播**：推给指定用户（收藏变动）
> - **时段推送**：指定时间段的人群

## 十、前端

### Q45：前端技术栈？
> React 18 + Vite + React Router + 自定义 CSS（无 UI 库）。

### Q46：页面性能怎么优化的？
> - **虚拟列表**：千篇文章首屏流畅
> - **图片懒加载**：`loading="lazy"`
> - **缓存**：API 响应缓存 5 分钟
> - **预加载**：hover 到卡片时预读取详情

### Q47：前端怎么鉴权？
> 从 localStorage 取 JWT token，`Authorization: Bearer xxx` 传给后端。401 时跳登录页。demo 用户发 `demo-user` 字符串。

### Q48：前后端怎么通信的？
> Nginx 反向代理，`/api/*` 转发到 FastAPI:8000，`/ws/*` 转发到 WebSocket。前端相对路径请求，无跨域问题。

### Q49：怎么部署的？
> **GitHub Actions 自动部署**。推送到 master → SCP 同步代码到服务器 → SSH 执行 `deploy.sh`（migration + pip install + npm build + 重启后端）。

### Q52：数据库 migration 怎么做？

## 十二、容错与稳定性

### Q53：整体容错策略总结？
| 层 | 容错方式 |
|----|---------|
| 采集 | 每个源独立 try/except，一个挂了不影响其他 |
| 去重 | AI 失败时标记 suspected_duplicate，不丢数据 |
| AI 处理 | 指数退避重试 3 次，耗尽后填默认值 |
| 缓存 | Redis 挂了自动降级到数据库直查 |
| JWT | Supabase 验证失败降级到直接解码 JWT |
| API | FastAPI ExceptionHandler 统一返回 JSON 错误 |
| 数据库 | 重试 3 次 + 重新创建客户端 |

### Q54：有没有遇到生产问题？怎么解决的？
> **场景**：Redis 因内存不足挂了，前端用户无法加载首页。  
> **分析**：CacheService 初始化时 Redis 连接失败，抛出异常导致启动失败。  
> **修复**：构造方法加 try/except，失败时 `available = False`，所有方法降级。  
> **学到的**：**外部依赖绝对不能阻塞核心流程。**

### Q55：SSE 资源泄漏问题？
> **场景**：用户刷新页面导致 SSE 连接断开，但 SseEmitter 没被清理，线程池泄漏。  
> **修复**：注册 `onCompletion()`、`onTimeout()`、`onError()` 回调，统一清理资源。  
> **学到的**：**异步连接必须三件套（完成/超时/错误）回调全处理。**

## 十三、优化与扩展

### Q56：如果文章量翻 10 倍，怎么扩展？
> 1. **消息队列**：RabbitMQ 异步解耦采集和 AI 处理
> 2. **Elasticsearch**：替代 SQL LIKE 搜索
> 3. **多进程部署**：uvicorn workers=4
> 4. **Redis 缓存层级**：热门文章直接缓存，不用 AI 处理
> 5. **前端虚拟滚动**：千篇以上无压力

### Q57：如果让你重新设计，会改什么？
> **"采集和 AI 处理解耦。** 当前是同步管道，采集完才能 AI 处理。引入消息队列后，采集完扔队列就走，AI 处理异步消费，这样采集频率可以更高，AI 处理失败也不阻塞下一轮采集。"

### Q58：最自豪的设计是什么？
> **"容错分层。** 每一层都有兜底机制，任何一个环节挂了都不会导致整个系统不可用。采集挂了跳过源，AI 挂了填默认值，缓存挂了降级查库，JWT 挂了降级解码。生产环境运行几个月，从没出现过完全不可用的情况。"

## 十四、场景题（面试官最爱的追问）

### Q59：如果用户反馈某天日报重复文章很多，怎么排查？
> 1. 查当天采集日志：看采集了哪些源、多少篇
> 2. 查去重日志：看 URL 去重和标题去重各过滤了多少
> 3. 查 AI 去重结果：AI 判断了哪些模糊配对
> 4. **大概率原因**：某个 RSS 源挂了，全文抓取返回了相似内容

### Q60：如果 AI 对话回答质量突然下降，怎么排查？
> 1. 检查 DeepSeek API 是否正常
> 2. 查缓存 key 是否命中（可能返回旧缓存）
> 3. 查上下文是否被撑爆（对话历史太长）
> 4. 查问题预分类是否误判（日报问题被分到通用知识，没注入上下文）

### Q61：如果上线新功能，怎么保证现有功能不受影响？
> 当前没有自动化测试覆盖。**临时方案**：手动跑一遍全流程 pipeline，确认采集→去重→AI→入库→前端展示链路正常。**长期方案**：加 CI 单元测试 + staging 环境。

### Q62：跨域问题遇到过吗？
> 开发时遇到过。前端 `localhost:5173` → 后端 `localhost:8000`。**解法**：Vite 配置 proxy 代理 `/api` 到 `http://localhost:8000`。生产环境用 Nginx 统一端口，不存在跨域。

## 十五、开放性问题

### Q63：这个项目给你最大的收获是什么？
> **"完整地走了一遍从 0 到 1 的全栈项目开发流程。** 最大的收获不是具体技术，而是**容错意识和成本意识**——每一层都要考虑失败了怎么办，每一个设计都要算成本账。"
