### 1. JWT 认证

#### 后端验证 — `api/services/jwt_verify.py`

```python
"""
认证链路（严格模式，P0 修复后）：
  1. demo-user → 返回 demo UUID（仅未登录浏览公开内容）
  2. JWT → 从 Supabase JWKS 获取公钥验证签名 → 返回 sub
  3. 全部失败 → 返回 None

  不再接受裸 UUID 作为认证凭证。
"""
import jwt
from jwt import PyJWKClient

SUPABASE_URL = os.getenv("SUPABASE_URL")
JWKS_URL = f"{SUPABASE_URL}/.well-known/jwks.json"

# 缓存 JWKS 客户端，1 小时重新获取一次
_jwks_client: Optional[PyJWKClient] = None
_jwks_cache_time = 0
_JWKS_TTL = 3600

def _get_jwks_client() -> PyJWKClient:
    global _jwks_client, _jwks_cache_time
    now = time.time()
    if _jwks_client is None or now - _jwks_cache_time > _JWKS_TTL:
        _jwks_client = PyJWKClient(JWKS_URL, cache_keys=True)
        _jwks_cache_time = now
    return _jwks_client


def verify_token(token: str) -> Optional[str]:
    if not token:
        return None
    if token.startswith("Bearer "):
        token = token[7:]

    # 1. demo 用户（仅未登录浏览）
    if token == DEMO_USER_ID:
        return DEMO_USER_UUID

    # 2. JWT 验证 — 唯一认证入口（P0 修复：删除了 UUID 直通）
    try:
        jwks_client = _get_jwks_client()
        signing_key = jwks_client.get_signing_key_from_jwt(token)
        decoded = jwt.decode(
            token,
            signing_key.key,
            algorithms=["RS256"],
            audience="authenticated",
            options={"verify_exp": True},
        )
        user_id = decoded.get("sub")
        if user_id:
            return user_id
    except jwt.ExpiredSignatureError:
        pass  # 过期
    except jwt.InvalidAudienceError:
        pass  # audience 不对
    except Exception:
        pass  # 签名不对

    return None  # 验证失败
```

**逐行讲解：**
- `PyJWKClient` 自动从 `/.well-known/jwks.json` 下载 RSA 公钥
- `cache_keys=True` + `_JWKS_TTL = 3600`：公钥缓存 1 小时，不用每次验证都 HTTP 请求
- `get_signing_key_from_jwt(token)`：从 JWT header（`kid` 字段）匹配对应公钥
- `jwt.decode(..., algorithms=["RS256"])`：RS256 是非对称签名，私钥签、公钥验，没人能伪造
- `audience="authenticated"`：只接受发给 Supabase Auth 的 token
- `options={"verify_exp": True}`：强制检查过期时间

**P0 修复前历史坑：**

```python
# 旧代码（危险！）：

```

```javascript
/** 预判过期（P2）：提前 5 分钟算即将过期 */
export function isTokenExpiringSoon() {
    const payload = JSON.parse(atob(token.split('.')[1]));
    return (payload.exp * 1000 - Date.now()) < 5 * 60 * 1000;
}

/** 自动刷新（P1）：调 Supabase SDK refreshSession 换新 token */
export async function refreshAccessToken(supabase) {
    const { data } = await supabase.auth.refreshSession();
    if (data?.session?.access_token) {
        setToken(data.session.access_token);
        return data.session.access_token;
    }
    return null;
}
```

#### 前端 401 拦截器 — `frontend/src/api/client.js`

```javascript
async function request(path, options = {}) {
    // 1. 预判：token 快过期 → 提前刷新
    if (isLoggedIn() && isTokenExpiringSoon()) {
        await refreshAccessToken(supabase);
    }

    let res = await fetch(...);

    // 2. 401 拦截（P1）：尝试刷新后重试
    if (res.status === 401 && isLoggedIn()) {
        const newToken = await refreshAccessToken(supabase);
        if (newToken) {
            res = await fetch(...);     // 重试原请求
        } else {
            clearToken();
            window.location.href = '/login';  // 跳登录
        }
    }

    if (res.status === 429) throw new Error('请求过于频繁');
    return res.json();
}
```

#### 面试官常见追问

| 问题 | 回答 |
|------|------|
| JWT 三段结构 | header(base64) . payload(base64) . signature。header 含算法 kid，payload 含 sub/exp/aud/iat |
| RS256 vs HS256 | RS256 公钥/私钥对，私钥签、公钥验，所有人都能验但不能伪造。HS256 对称密钥，能验的人也能伪造 |
| refresh token 为什么需要？ | access_token 1 小时过期，refresh_token 长期有效。用户操作中断时无感续期 |
| 攻防：token 被偷了 | 改进方向：refresh token rotation（每次刷新发新 refresh token 废掉旧的）|
| 攻防：暴力破解密码 | 登录接口 Supabase Auth 管理，我们后端做的 `/api/auth/` 接口单独 30/min 限流 |
| 为什么 UUID 直通是漏洞？ | UUID 不是秘密（很多 API 响应里会暴露 user_id），如果有人拿到了就能冒充 |

### 2. XSS 防御 — `frontend/src/utils/markdown.js` L45-95

```javascript
// 第一步：白名单 — 只允许这些 HTML 标签
const allowed = new Set([
    'p', 'br', 'h1', 'h2', 'h3', 'h4',
    'ul', 'ol', 'li',
    'strong', 'b', 'em', 'i', 'u',
    'blockquote', 'pre', 'code', 'span', 'div',
    'hr',
]);
```

**为什么需要这套东西？**
AI 对话回复里用的是 Markdown，要渲染成 HTML 展示给用户。但 LLM 可能被 prompt injection 诱导输出 `<script>alert('xss')</script>`。不能直接 `innerHTML`。

**处理流程（L48-89）：**
```javascript
function sanitizeHtml(html) {
    // 第 1 步：先把所有 &<> 转义成 HTML 实体
    let safe = html
        .replace(/&(?!...)/g, '&amp;')  // & → &amp;
        .replace(/</g, '&lt;')          // < → &lt;
        .replace(/>/g, '&gt;');         // > → &gt;
    
    // 第 2 步：正则匹配 "&lt;strong&gt;" 这种被转义后的标签
    // 如果标签在白名单里，恢复成真正的 <strong>
    // 如果不在白名单里（如 script），直接删除
    const tagPattern = /&lt;(\/?)(\w+)([^&]*)&gt;/g;
    safe = safe.replace(tagPattern, (match, slash, tagName, attrs) => {
        const tag = tagName.toLowerCase();
        if (allowed.has(tag)) {
            return `<${slash}${tag}>`;   // 白名单 → 恢复
        }
        return '';                        // 非白名单 → 删掉
    });
    
    // 第 3 步：再把 &amp; 之类的实体解码回可读形式
    safe = safe.replace(/&amp;/g, '&')
               .replace(/&lt;/g, '<')
               ...;
    return safe;
}
```

**第 2 层防线 — DOMPurify：**

`frontend/src/components/ArticleReader.jsx` L378-382：

2. `DOMPurify.sanitize()` — 业界标准库兜底

### 3. SQL 索引 — `scripts/init_schema.sql`

```sql
-- 文章表
CREATE TABLE articles (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title TEXT NOT NULL,
    url TEXT UNIQUE NOT NULL,
    source_name TEXT NOT NULL,
    tags TEXT[],
    published_at TIMESTAMPTZ,
    ...
);

-- 索引
CREATE INDEX idx_articles_published_at ON articles(published_at DESC);
-- 原因：日报按时间排序，每次首页都要 ORDER BY published_at DESC

CREATE INDEX idx_articles_tags ON articles USING GIN(tags);
-- 原因：tags 是 TEXT[] 数组字段，普通 B-tree 不能索引数组
-- GIN 是 PostgreSQL 专门索引数组/JSONB 的索引类型
-- 查询时 WHERE tags @> ARRAY['AI'] 才能走索引

CREATE INDEX idx_articles_source ON articles(source_name);
-- 原因：用户按来源筛选时 WHERE source_name = '机器之心'

-- 用户收藏表
CREATE TABLE bookmarks (
    user_id UUID NOT NULL,
    article_id UUID NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_bookmarks_user ON bookmarks(user_id, created_at DESC);
-- 重点：复合索引 (user_id, created_at)
-- user_id 先过滤出当前用户，再按 created_at 排序
-- 如果只建 user_id 索引，排序还是要全表扫描
```

```python
async def upload_document(
    file: UploadFile = File(...),
```

**失败重试 — `api/models/database.py` L50-62：**
```python
def _execute_with_retry(self, operation, *args, **kwargs):
    last_error = None
    for attempt in range(self.MAX_RETRIES + 1):  # 最多 4 次（0,1,2,3）
        try:
            return operation(*args, **kwargs)
        except Exception as e:
            last_error = e
            if attempt < self.MAX_RETRIES:
                # 退避等待：2s, 4s, 8s
                wait = self.RETRY_DELAY * (2 ** attempt)
                time.sleep(wait)
                # 重新创建客户端（可能连接断了）
                self.client = self._create_client()
    raise last_error
```

### 3. 分页 — `api/models/database.py` L82-98

```python
def get_articles(self, page=1, page_size=20, keyword=None, ...):
    query = self.client.table("articles").select("*", count="exact")
    
    # 过滤条件链
    if keyword:
        query = query.or_(f"search_vector.phfts.{keyword},...")
    if tag:
        query = query.contains("tags", [tag])
    if source:
        query = query.eq("source_name", source)
    
    # 排序
    query = query.order("published_at", desc=True)
    
    # 分页：第 2 页 20 条 → range(20, 39)
    offset = (page - 1) * page_size
    query = query.range(offset, offset + page_size - 1)
    
    result = query.execute()
    total = result.count or 0
    
    return {
        "items": result.data,
        "total": total,
        "page": page,
        "page_size": page_size,
        "pages": (total + page_size - 1) // page_size,
    }
```

```
VITE_SUPABASE_ANON_KEY=eyJ...  # anon key，只有最低权限
```
`VITE_` 前缀是 Vite 约定的，只有带这个前缀的变量才会暴露给前端代码。

**为什么 anon key 可以暴露给前端？** Supabase 在数据库层面做了 RLS（Row Level Security）——即使有 anon key，也只能读写自己的数据，改不了别人的。

## 附录：面试官问 JWT 认证，你按这个节奏讲

> 把下面这段话**用嘴说一遍**，流畅度合格了底气就有了。

### "能讲讲你们的认证系统吗？"

**第一段（30 秒 · 链路概览）：**

"我们用的是 Supabase Auth + JWT。流程是这样的：

用户登录时前端调 Supabase SDK 的 `signInWithPassword()`，Supabase 验证邮箱密码后返回三个东西：`access_token`、`refresh_token`、`user`。前端把 `access_token` 存到 localStorage。后续每次请求，API 客户端 `client.js` 从 localStorage 读 token，拼成 `Authorization: Bearer <token>` 发到后端。后端收到后，用 `jwt_verify.py` 里的 `verify_token()` 函数验证签名，验证通过就从 token 的 `sub` 字段里取出用户 ID，注入到路由里。"

**第二段（30 秒 · 签名验证细节）：**

"验证方式用的是 RS256 + JWKS。Supabase 暴露了一个 `/.well-known/jwks.json` 端点，里面放着 RSA 公钥。后端的 `PyJWKClient` 会去拉这个公钥，缓存 1 小时。验证时从 JWT 的 header 里找到 `kid` 字段匹配公钥，然后用 RS256 验签。还有人记得我有个历史坑——最开始我跳过了签名验证，写成 `verify_signature=False`，任何人都能伪造 token 冒充别人，上线两天发现了赶紧改成了 JWKS 公钥验证。"

**第三段（20 秒 · Token 自动刷新）：**

"access_token 默认 1 小时过期。我们在前端做了两层保护：

- 第一层是预判——发请求前检查 token 的 `exp` 字段，如果剩余时间不到 5 分钟，先用 `supabase.auth.refreshSession()` 换新 token 再发请求。
- 第二层是 401 拦截——如果 token 已经过期了，后端返回 401。前端拦截到这个状态，自动调 refresh，拿到新 token 后重试原请求。如果刷新也失败了（比如 refresh_token 也过期了），清除本地 token，跳转到登录页。

整个过程用户完全无感知。我从 Chrome DevTools 的 Application → Local Storage 里看到 token 在静默更新。"

**第四段（10 秒 · 安全加固）：**

"还有两个加固点：一是后端把裸 UUID 直接通过认证的漏洞封了，所有非 demo 用户的请求必须走 JWT 签名验证；二是对收藏、历史这些用户操作接口单独做了 30 次/分钟的限流，防止被刷。"

"总分结构" **面试官可能接着问：**

| 追问 | 你接 |
|------|------|
| "JWT 三段结构是啥？" | header.payload.signature。header 是 base64 编码的算法信息，payload 是用户信息和过期时间，signature 用私钥签名防篡改。可以在 jwt.io 上解码看。 |
| "为什么 RS256 不是 HS256？" | RS256 是非对称的，私钥只有 Supabase 有，公钥可以公开。所有人都能验证但没人能伪造。HS256 是对称的，会签名的也能伪造。 |
| "refresh token 有风险吗？" | 有。如果 refresh_token 被盗了，攻击者能长期持有登录状态。改进方案是 refresh token rotation — 每次刷新发新 token 的同时废弃旧的。 |
| "你们和标准的 OAuth2 流程有啥区别？" | 我们用 Supabase 封装了，没有手写授权码流程。相当于 Supabase 同时做了授权服务器和令牌端点。我们直接拿到 JWT 就用了。 |
