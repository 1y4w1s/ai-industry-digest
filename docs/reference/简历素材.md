## 三、功能矩阵（按技术深度排序）

### 3.1 核心竞争力

| 功能 | 实现细节 | 技术难点 |
|------|---------|---------|
| **自动化采集→AI→日报全链路** | GitHub Actions cron 触发 → collector(httpx+feedparser) → AIProcessor(DeepSeek) → reporter → Supabase | 多源去重、AI 批处理优化、失败重试 |
| **AI 个性化推荐** | TagExtractor(零API成本字符串匹配) → user_tags 画像(阅读/对话/收藏) → recommend.py 排序 → 前端 RecommendationWidget | 本地匹配 vs AI 调用平衡 |
| **知识图谱** | 上传→切片(chunking)→实体识别+关系抽取(DeepSeek)→力导向图可视化(Canvas) | 大文件分片、图谱去重、增量更新 |
| **公共知识库+权限** | 上传时公开/私有切换，`_doc_access_filter` 统一权限模型(OR 逻辑) | 权限查询不破坏索引性能 |
| **全站搜索** | `/api/search` 同时搜 articles + kb_documents，前端搜索结果合并展示 | 异构数据统一分页 |

## 四、安全清单（面试加分点）

| 措施 | 实现 |
|------|------|
| JWT 签名验证 | 从 Supabase JWKS 端点获取公钥，RS256 验证 |
| CORS | 白名单域名（非通配符 `*`） |
| 限流 | 内存限流（IP 上限 1000 + 自动清理） |
| XSS | DOMPurify 清洗 AI 回复 HTML |
| 路径穿越 | `EXTENSION_MAP` 服务端确定扩展名 |
| 认证一致性 | 统一 `verify_token()` 支持 Header + Query token |
| 数据库连接 | 单例 `get_db()` 防连接池耗尽 |
| SQL 注入 | ORM 查询（Supabase JS SDK） |

## 六、STAR 话术模板（直接拿来喂 AI）

### 场景 1：全栈项目

> **S**: 需要搭建一个 AI 行业资讯平台，实现从信息采集到用户阅读的全自动化闭环。  
> **T**: 单人独立完成从前端到后端到部署的全栈开发，技术栈横跨 React/FastAPI/Supabase/DeepSeek API。  
> **A**: 实现每日 3 次自动采集 → AI 摘要生成 → 日报入库的全自动流水线；开发了知识库管理系统（上传/预览/图谱/权限），日均处理 400+ 篇文章；建立 41 项单元测试保障质量。  
> **R**: 平台稳定运行于公网，全自动无人值守。后期安全性大重构修复 15 项问题（JWT 验签/CORS/路径穿越/XSS）。

### 场景 2：AI 集成

> **S**: 需要将 LLM 能力集成到传统 Web 应用中，且不能每次都调 API 导致成本失控。  
> **T**: 设计了一个混合 AI 架构——高频简单操作(标签提取)用本地字符串匹配零成本，复杂任务(摘要/实体)才调 DeepSeek API。  
> **A**: 实现 TagExtractor 本地匹配 + user_tags 画像个性化推荐 + 异步后台标签提取不阻塞主请求。  
> **R**: 使每个用户获得个性化推荐体验，API 调用成本降低约 60%。

### 场景 3：安全重构

> **S**: 项目初期仅 Demo 认证(跳过JWT验签)，存在严重安全风险。  
> **T**: 需要在不影响现有功能的前提下引入完整认证链路。  
> **A**: 全面审计认证链路，实现 JWKS 公钥验签、CORS 白名单、Service Key 清理、路径穿越防御等 15 项修复。  
> **R**: 安全评分从 2/10 提升至 8/10，用户数据得到生产级保护。
