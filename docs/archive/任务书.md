## P0 — 安全漏洞（即刻修复，不修就危）

| # | 问题 | 文件 | 风险 |
|---|------|------|------|
| 0.1 | JWT 不做签名验证 — 任何人可伪造 token 冒充任意用户 | `api/routes/auth.py` L34 | 数据泄露 |
| 0.2 | Supabase Service Role Key 硬编码在脚本中（已 git 提交） | `scripts/migrate_kb_user.py`（已删除但 git history 中有） | 数据库沦陷 |
| 0.3 | 文件上传 `ext` 来自用户控制的 `file.filename`，可能路径穿越 | `api/routes/kb.py` L119 | 任意文件覆盖 |

### 0.1 JWT 签名验证
- 从 Supabase 获取 JWKS 公钥，`jwt.decode(..., algorithms=["RS256"])`
- 影响路由：`auth.py` 中 `get_user_id()` / `kb.py` 中 `_resolve_user_id()`

### 0.2 Service Key 清理
- 确认 `scripts/migrate_kb_user.py` 和 `scripts/list_users.py` 已删除
- 在 Supabase 控制台 rotation 当前 service role key
- 今后所有迁移脚本从 `.env` 中读取 key

### 0.3 路径穿越修复
- 文件保存路径完全由后端控制：`f"{document_id}{SUPPORTED_EXTENSIONS[file_type]}"`
- 不使用用户文件名后缀

## P1 — 架构缺陷（必须修）

| # | 问题 | 文件 | 影响 |
|---|------|------|------|
| 1.1 | 无连接池，每次请求新建 DatabaseManager | `api/models/database.py` | 并发 → 连接数耗尽 → 502 |
| 1.2 | 首页 `/api/home` 一次性返回全部数据 | `api/routes/content.py` | 数据量大时爆炸 |
| 1.3 | 速率限制用内存字典，多进程无效 | `api/main.py` L37 | 限流形同虚设 |
| 1.4 | 部署无自动 migration | `scripts/deploy.sh` | 改表结构需人工 SQL |

### 1.1 DatabaseManager 单例化
```python
# database.py 底部
_db_manager = None
def get_db() -> DatabaseManager:
    global _db_manager
    if _db_manager is None:
        _db_manager = DatabaseManager()
    return _db_manager
```
或使用 `@classmethod` / `__new__` 单例模式。

- 所有路由文件 `db = DatabaseManager()` → `db = get_db()`

### 1.2 首页分页
- `GET /home` 改为只返回日报元数据（标题+日期），日报详情由前端按需请求 `GET /reports/{date}`

### 1.3 速率限制改进
- 迁移到 Nginx 限流
- 或改用文件/共享内存，至少加内存上限清理

### 1.4 自动 migration 脚手架
- 创建 `scripts/migrate.py`，读取 `scripts/migration_*.sql` 按次序执行
- `deploy.sh` 中 `python scripts/migrate.py` 作为第一步

## P2 — 安全强化 + 性能优化

| # | 问题 | 文件 | 说明 |
|---|------|------|------|
| 2.1 | CSRF 防护缺失，CORS allow_origins=["*"] | `api/main.py` | 低风险但合规需要 |
| 2.2 | 前端 1.2MB 全量 JS 不分 chunk | `frontend/src/App.jsx` | 首屏加载慢 |
| 2.3 | Token 管理分散，`localStorage` key 硬编码在 2 处 | `AuthContext.jsx` + `client.js` | 不同步风险 |
| 2.4 | 无集成测试，被测模块覆盖率低 | `tests/` | 回归无法发现 |
| 2.5 | 前端 `dist/` 被 git 追踪 | `.gitignore` | 反模式 |

### 2.1 CSRF 防护
- 将 `allow_origins=["*"]` 收紧为白名单域名
- 对写操作（POST/PUT/DELETE）校验 `Origin` 或 `Referer`

### 2.2 代码分割
```jsx
const KnowledgeBasePage = React.lazy(() => import('./pages/KnowledgeBasePage'));
const ProfilePage = React.lazy(() => import('./pages/ProfilePage'));
// ... 等所有非首屏页面
```

### 2.3 统一 Token Service
- `src/lib/auth.js` 导出 `getToken()` / `setToken()` / `clearToken()`
- `AuthContext` 和 `api/client.js` 都引用同一个模块

### 2.4 测试补全
- 为所有 API 路由添加 `TestClient` 集成测试
- 为首页、推荐、知识库关键链路添加 E2E 测试
- 数据库 migration 测试（建表 → 查询 → 删表）

### 2.5 清理 gitignore
- `frontend/dist/` 加入 `.gitignore` 并 git rm

## P3 — 代码质量 + 体验优化

| # | 问题 | 文件 | 说明 |
|---|------|------|------|
| 3.1 | 表名硬编码在所有路由中 | `api/routes/kb.py` 等 | 表名变更需全量替换 |
| 3.2 | TagExtractor 传递字符串而非 ID 列表 | `chat.py` background_tasks | 大列表时低效 |

```python

class Tables:
    ARTICLES = "articles"
    KB_DOCUMENTS = "kb_documents"
    KB_CHUNKS = "kb_chunks"
    ...
```

### 3.2 TagExtractor 优化
- 改为传 `article_ids: List[str]`，函数内部批量查询 tags

### 3.3 全站搜索
- 搜索框请求 `/api/search?q=xxx` 同时搜 articles + kb_documents

### 3.4 健康检查端

```python
async def health():
    db = get_db()
    db.client.table("articles").select("id").limit(1).execute()
    return {"status": "ok", "db": "ok", "version": "2.0.0"}
```

### 3.5 骨架屏补全
