## 根因分析

### 直接原因：数据库缺少字段

知识库上传接口的代码中，INSERT/UPDATE 操作使用了以下字段，但数据库表中没有这些列：

| 缺失字段 | 用途 | 影响的操作 | 所属迁移文件 |
|---------|------|-----------|------------|
| `is_public` | 标记文档是否公开 | `INSERT INTO kb_documents` | `scripts/migration_kb_public.sql` |
| `content_hash` | 文档内容 MD5 哈希（去重/增量更新） | `UPDATE kb_documents` | `scripts/migration_incremental_update.sql` |
| `version` | 文档版本号（增量更新） | `UPDATE kb_documents` | `scripts/migration_incremental_update.sql` |

### 调用链路

```
[前端] handleUpload()
  → POST /api/kb/documents (FormData)
    → [kb.py] upload_document() 
      → db.client.table("kb_documents").insert({ is_public, ... })   ← ❌ is_public 列不存在
        → Supabase 抛出异常
          → FastAPI 返回 500 Internal Server Error
```

### 根本原因：部署流程缺陷

`scripts/deploy.sh` 第 31 行执行 migration 时，错误输出被吞掉：

```bash
python scripts/migrate.py 2>/dev/null    # ← 错误信息被丢弃
```

而 `scripts/migrate.py` 通过 Supabase REST API 执行 `ALTER TABLE` 这类 DDL 语句，**权限不足导致静默失败**。

---

## 修复方案

### 修复操作

登录 Supabase 控制台 → SQL Editor，依次执行：

```sql
-- 迁移 1: 添加 is_public 字段
ALTER TABLE IF EXISTS kb_documents 
ADD COLUMN IF NOT EXISTS is_public BOOLEAN DEFAULT TRUE;

-- 迁移 2: 添加 content_hash 和 version 字段
ALTER TABLE IF EXISTS kb_documents 
ADD COLUMN IF NOT EXISTS version INTEGER DEFAULT 1;
ALTER TABLE IF EXISTS kb_documents 
ADD COLUMN IF NOT EXISTS content_hash VARCHAR(64) DEFAULT '';
```

### 部署脚本改进（建议）

修改 `scripts/deploy.sh` 第 31 行，去掉错误重定向：

```diff
- python scripts/migrate.py 2>/dev/null && echo "   ✅ Migration 完成" || echo "   ⚠️ 部分 migration 需人工执行"
+ python scripts/migrate.py && echo "   ✅ Migration 完成" || echo "   ⚠️ Migration 失败，请手动执行 SQL（见 scripts/migration_*.sql）"
```
