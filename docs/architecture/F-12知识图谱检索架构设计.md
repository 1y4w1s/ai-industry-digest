# F-12 知识图谱检索架构设计

## 1. 概述

### 1.1 背景

RAG 系统的检索层当前依赖**两路信号融合**：

| 信号 | 能力 | 盲区 |
|------|------|------|
| 向量检索（语义相似度） | 理解查询语义，泛化能力强 | 不关心实体之间的显式关系 |
| 关键词检索（精确匹配） | 精确命中关键词 | 无法理解"实体 A 与实体 B 有什么关联" |

当用户查询涉及**实体关系**（如"OpenAI 的投资方"、"GPT 和 BERT 区别"）时，上述两路信号都不够充分。需要引入**第三路信号**——知识图谱检索——利用已提取的实体-关系网络来增强检索召回。

### 1.2 目标

- 在现有 RRF 两路融合（向量 + 关键词）基础上，增加**图谱检索**作为第三路信号
- 对涉及实体关系的查询，能够从知识图谱中检索到高质量的相关文档切片
- 所有降级路径无损：图谱服务不可用时系统照常工作

## 2. 系统架构

### 2.1 位置

```
用户查询
    │
    ▼
AdvancedRetrievalService.search()
    │
    ├── (1) 意图路由  ──→  QueryRouterService
    │
    ├── (2) Query改写  ──→  rewrite_query()
    │
    ├── (3) 三路并行检索
    │       ├── 向量检索  ──→  vector_search()
    │       ├── 关键词检索 ──→  keyword_search()
    │       └── 图谱检索  ──→  GraphRetrievalService    ★ 新增
    │
    ├── (4) RRF 三路融合 ──→  rrf_fusion(v, k, g, k=60)
    │                        权重: 0.50 : 0.30 : 0.20
    │
    └── (5) Re-ranker 精排 ──→  RerankerService
```

### 2.2 数据流

```
         ┌──────────────┐
         │  用户查询     │
         │ "OpenAI投资方" │
         └──────┬───────┘
                │
                ▼
      ┌─────────────────┐
      │ jieba 分词      │ → ["OpenAI", "投资方"]
      │ 提取候选实体    │
      └────────┬────────┘
               │
               ▼
      ┌─────────────────┐
      │ kb_entities 模糊 │ → ilike "%OpenAI%"

      └────────┬────────┘
               │
               ▼
      ┌─────────────────┐
      │ kb_chunks 关联   │ → WHERE document_id IN (实体所属文档)
      │ 获取切片        │
      └────────┬────────┘
               │
               ▼
      ┌─────────────────┐
      │ Graph Score 计算 │ → entity_hits + relation_boost
      │                  │ → score ∈ [0, 1.0]
      └────────┬────────┘
               │

```

## 3. 核心算法

### 3.1 GraphRetrievalService.graph_search()

```
输入: query(str), user_id(str), limit(int=5)
输出: List[{chunk, document, score, graph_info}]

流程:
  1. 实体提取
      query  ──jieba分词──►  候选实体列表
      过滤停用词和单字
  
  2. 数据库实体匹配
      对每个候选实体 → kb_entities.ilike("%{candidate}%")
      去重 → 匹配实体列表 [{id, name, type, document_id}]
  
  3. 切片检索
      获取匹配实体的 document_id 集合
      → kb_chunks WHERE document_id IN (doc_ids)
      → 过滤权限（公开 OR 用户自有）
      → 按时间排序，取 limit*3 条
  
  4. 分数计算
      for each chunk:
          entity_hits = count(entity_name in chunk_content)
          entity_score = min(entity_hits / total_entities * 2, 1.0)
          relation_boost = 0.2 if is_relation_query(query) else 0.0
          score = min(entity_score + relation_boost, 1.0)
  
  5. 排序输出
      按 score 降序 → 取前 limit 条
```

### 3.2 RRF 三路融合

将原始的两路 RRF 升级为三路，权重设计原则：

| 信号 | 权重 | 理由 |
|------|------|------|
| 向量检索 | 0.50 (50%) | 语义匹配能力强，覆盖面最广 |
| 关键词检索 | 0.30 (30%) | 精确匹配，弥补向量对罕见词的不敏感 |
| 图谱检索 | 0.20 (20%) | 实体关系信息，仅对涉及实体的查询有效 |

公式：

```
fused_score = 0.50 * (1 / (k + vector_rank)) +
              0.30 * (1 / (k + keyword_rank)) +
              0.20 * (1 / (k + graph_rank))
```

其中 `k=60`（RRF 常数），`graph_rank = inf` 时该项为 0（向后兼容）。

**向后兼容性**：不传 `graph_results` 参数时，自动降级为两路融合（0.60 * vector + 0.40 * keyword）。

### 3.3 关系查询检测

通过正则表达式模式匹配识别意图为"关系查询"的请求，这类查询获得 0.2 的图谱分数增强：

```
模式清单（按优先级）:
  ┌ "与/和 ... 的关系/关联/相关"
  ├ "... 的投资方、创始人、产品、母公司"
  └ "related to / connected to"
```

## 4. 数据库表结构

涉及三张表（无新增）：

### kb_entities（实体表）

```sql
CREATE TABLE kb_entities (
    id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id UUID REFERENCES kb_documents(id) ON DELETE CASCADE,
    name       VARCHAR(255) NOT NULL,
    type       VARCHAR(50),      -- concept, person, organization, technology
    created_at TIMESTAMP DEFAULT NOW()
);
```

### kb_relations（关系表）

```sql
CREATE TABLE kb_relations (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id       UUID REFERENCES kb_documents(id) ON DELETE CASCADE,
    source_entity_id  UUID REFERENCES kb_entities(id) ON DELETE CASCADE,
    target_entity_id  UUID REFERENCES kb_entities(id) ON DELETE CASCADE,
    relation_type     VARCHAR(100),  -- is_a, part_of, related_to, based_on
    label             VARCHAR(100),  -- 中文标签（如"开发"、"投资"）
    created_at        TIMESTAMP DEFAULT NOW()
);
```

### kb_chunks（切片表 — 融合输出目标）

```sql
CREATE TABLE kb_chunks (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id UUID REFERENCES kb_documents(id) ON DELETE CASCADE,
    content     TEXT NOT NULL,
    chunk_index INTEGER,
    metadata    JSONB,      -- 含 extracted_entities 字段
    created_at  TIMESTAMP DEFAULT NOW()
);
```

### 关联查询路径

```
查询 "OpenAI投资方"
    │
    ├── 实体匹配: kb_entities.name ilike '%OpenAI%'
    │   └─→ id = "uuid-xxx"
    │
    ├── 找到所属文档: kb_entities.document_id = "doc-123"
    │
    └── 获取切片: kb_chunks WHERE document_id = "doc-123"
        └─→ 匹配切片列表
```

## 5. 降级策略

| 场景 | 表现 | 用户感知 |
|------|------|----------|
| 数据库连接失败 | try/catch 捕获异常，跳过图谱检索 | 检索结果中缺少图谱信号，精度略有下降 |
| jieba 分词异常 | 实体候选为空 → graph_search 返回空列表 | 同上 |
| 实体表无数据（新系统） | 无匹配实体 → 返回空列表 | 同上 |
| 关键词检索/向量检索正常 | RRF 两路融合正常工作 | 无感知 |
| 所有三路均异常 | search() 返回空列表 | 无结果（但 Reranker/压缩正常） |

关键设计原则：**图谱检索永远作为一个"可选的辅助信号"**，任何异常都被捕获并静默降级，不影响主流程。

## 6. 接口定义

### 6.1 GraphRetrievalService

```python
class GraphRetrievalService:
    
    def graph_search(
        self,
        query: str,          # 用户查询（已改写或原始）
        user_id: str,        # 用户 ID（用于权限过滤）
        limit: int = 5,      # 返回结果数
    ) -> List[Dict]:
        """基于知识图谱的检索"""
    
    def _extract_query_entities(self, query: str) -> List[str]:
        """从查询中提取候选实体名（jieba分词 + 停用词过滤）"""
    
    def _is_relation_query(self, query: str) -> bool:
        """检测是否是关系查询（正则模式匹配）"""
    
    def _find_matched_entities(self, candidates, user_id) -> List[Dict]:
        """在 kb_entities 中搜索匹配实体（ilike 模糊匹配）"""
    
    def _find_chunks_by_entities(self, entity_ids, entity_names, user_id, limit) -> List[Dict]:
        """根据实体查找关联文档切片"""
```

### 6.2 AdvancedRetrievalService 集成

```python
# 调用方代码
graph_service = get_graph_retrieval_service()
graph_results = graph_service.graph_search(query, user_id, limit)

# 三路融合
fused = self.rrf_fusion(vector_results, keyword_results, graph_results)
```

## 7. 与现有检索的对比

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 融合路数 | 2路（向量 + 关键词） | 3路（向量 + 关键词 + 图谱） |
| 检索范围 | 全部切片按语义/关键词匹配 | 增加实体关联的定向检索 |
| 关系查询理解 | 无法处理 | 可识别"投资方""关系"等模式 |
| 数据利用 | 仅利用切片内容 | 同时利用实体-关系知识网络 |
| RRF 权重 | 60:40 | 50:30:20 |
| 降级机制 | 单路失败不影响另一路 | 三路中任意一路失败自动降级 |
| 测试覆盖 | 无图谱相关测试 | 18 个单元测试（含关系检测/实体提取/边界） |

## 8. 已知局限与后续优化

| 局限 | 影响 | 后续改进方向 |
|------|------|-------------|
| 仅通过 `document_id` 关联实体和切片 | 粗粒度：同一文档中不相关的切片也被召回 | 引入 `kb_entity_chunks` 关联表，实现细粒度切片-实体绑定 |
| 实体匹配仅支持单表 `ilike` 模糊匹配 | 不支持同义词/缩写（如"OpenAI" vs "Open AI"） | 引入实体别名表或 Embedding 语义匹配 |
| Graph Score 为启发式公式 | 简单加权，未利用关系类型信息 | 加入关系类型权重（如"投资" > "相关"） |
| 未利用 `kb_relations` 的路径信息 | 无法做"OpenAI 投资了哪些公司"的多跳查询 | 实现关系路径遍历：entity → relation → entity → chunk |
| `_extract_query_entities` 依赖 jieba 词典 | 对专有名词（如"GPT-4"）分割可能不准确 | 引入自定义词典动态更新 |

## 10. 关键决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 图谱作为独立信号 vs 嵌入向量检索 | 二选一 | **独立信号** | 解耦设计，任一环节故障不影响其他路 |
| Graph Score 计算方式 | 启发式公式 vs 模型预测 | **启发式公式** | 零延迟、零外部依赖，适合作为辅助信号 |
| 实体关联文档方式 | document_id vs entity_chunks 关联表 | **document_id** | 当前 entity_chunks 表无数据，document_id 可立即生效 |
| 图谱权重 20% | 10%/20%/30% | **20%** | 太低无影响，太高会干扰语义信号；20% 可在不降低主指标的前提下补充信息 |
