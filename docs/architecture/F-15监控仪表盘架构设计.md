# F-15 检索质量监控仪表盘架构设计

## 1. 概述

### 1.1 背景

Phase 1 已在 `AdvancedRetrievalService` 中实现了基础的检索日志（`_log_search`），将每次查询的关键指标写入 `retrieval.log`。但目前缺乏：

- **聚合分析**：无法知道近期的平均检索精度、平均响应时间
- **可视化**：日志文件难以直观反映系统运行状态
- **告警**：没有异常检索行为的自动检测机制
- **对比评估**：无法对比不同路由策略/切片参数的效果差异

F-15 的目标是构建一个**轻量、可观测、可扩展**的监控体系，覆盖检索质量、系统性能、用户行为三个维度。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **零侵入** | 监控代码通过装饰器/中间件接入，不侵入检索核心逻辑 |
| **最小依赖** | 不使用外部监控系统（Prometheus/Grafana），数据库即仪表盘 |
| **可进化** | MVP 是 API-only 的 JSON 端点，可后期无缝对接前端 |
| **自包含** | 所有监控数据存储在 `kb_metrics` 一张表中 |

## 2. 指标体系

### 2.1 指标分类

```
检索质量指标（Quality）
├── precision@k       —— 前 k 个结果的相关性（通过用户点击隐式反馈）
├── recall@k          —— 前 k 个结果的覆盖率（通过 TopN 评分分布估算）
├── zero_result_rate   —— 无结果查询的比例
├── fusion_score_dist  —— RRF 融合分数分布
└── rerank_boost       —— Re-ranker 排序对 Top1 的改善幅度

系统性能指标（Performance）
├── p50/p95/p99 延迟    —— 检索全链路耗时（ms）
├── vector_search_ms    —— 向量检索耗时
├── keyword_search_ms   —— 关键词检索耗时
├── graph_search_ms     —— 图谱检索耗时
├── rerank_ms           —— 重排序耗时
├── compress_ms         —— 压缩耗时
├── total_tokens        —— 每次查询消耗的 tokens
└── cache_hit_rate      —— 结果缓存命中率（未来扩展）

路由分布指标（Routing）  
├── route_distribution   —— 各意图类型的查询占比
├── route_effectiveness  —— 各路由策略的 avg_top_score
└── route_latency        —— 各路由策略的平均延迟

压缩评估指标（Compression）
├── compression_ratio     —— 压缩前后字符数比例
├── avg_compressed_chars  —— 平均压缩后长度
└── compress_mode_dist    —— extract vs summarize vs truncate 分布
```

### 2.2 关键量定义

```
precision@5 = count(用户点击了检索结果中的前5条) / 5
  —— 依赖隐式反馈，需要前端配合记录点击事件

rerank_boost = (re_score_top1 - fused_score_top1) / fused_score_top1
  —— 正值表示 Re-ranker 改善了排序质量

compression_ratio = compressed_chars / original_chars
  —— 越小说明压缩越激进（< 0.4 可能丢失信息）

zero_result_rate = 返回0条结果的查询数 / 总查询数
  —— > 10% 需要排查切片/Embedding/权限问题
```

## 3. 数据存储

### 3.1 `kb_metrics` 表

单表存储所有监控指标，使用 JSONB 字段支持灵活的指标结构：

```sql
CREATE TABLE IF NOT EXISTS kb_metrics (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    metric_type     VARCHAR(32)   NOT NULL,  -- search / rerank / compress / route / error
    user_id         VARCHAR(64)   DEFAULT '',
    query           TEXT          DEFAULT '',
    timestamp       TIMESTAMP     DEFAULT NOW(),
    
    -- 核心指标（所有类型共用）
    latency_ms      INTEGER       DEFAULT 0,
    
    -- 检索指标
    vector_count    INTEGER       DEFAULT 0,
    keyword_count   INTEGER       DEFAULT 0,
    graph_count     INTEGER       DEFAULT 0,
    final_count     INTEGER       DEFAULT 0,
    top_scores      JSONB         DEFAULT '[]',  -- [0.95, 0.82, ...]
    route           VARCHAR(16)   DEFAULT '',     -- recommend / definition / comparison / time / general
    mode            VARCHAR(16)   DEFAULT '',     -- hybrid / vector_only / keyword_only / graph_only
    
    -- 重排序指标
    pre_rerank_top1_score   REAL  DEFAULT 0,
    post_rerank_top1_score  REAL  DEFAULT 0,
    rerank_delta            REAL  DEFAULT 0,     -- post - pre
    
    -- 压缩指标
    original_chars     INTEGER  DEFAULT 0,
    compressed_chars   INTEGER  DEFAULT 0,
    compress_mode      VARCHAR(16) DEFAULT '',
    
    -- 路由指标
    intent_type   VARCHAR(16)   DEFAULT '',
    limit_mult    REAL          DEFAULT 1.0,
    needs_time_filter BOOLEAN   DEFAULT FALSE,
    
    -- 附加值
    error_msg      TEXT          DEFAULT '',
    extra          JSONB         DEFAULT '{}'
);

-- 按类型和时间查询的索引
CREATE INDEX IF NOT EXISTS idx_metrics_type_ts ON kb_metrics (metric_type, timestamp DESC);
CREATE INDEX IF NOT EXISTS idx_metrics_user_ts ON kb_metrics (user_id, timestamp DESC);
```

### 3.2 数据量估算

| 指标 | 估算 |
|------|------|
| 每条查询产生的记录数 | 1 ~ 4 条（search + rerank + compress + route） |
| 日均记录数（1000查询） | 1000 ~ 4000 条 |
| 月均增长 | ~120,000 条 |
| 单条记录大小 | ~500 bytes |
| 一年数据量 | ~200 MB（可接受无清理） |

建议：保留 6 个月数据，超期自动归档。

## 4. 架构设计

### 4.1 整体架构

```
用户查询 → [MetricsMiddleware] → AdvancedRetrievalService.search()
                                       │
                              ┌────────┴────────┐
                              ▼                  ▼
                        vector_search()    keyword_search()
                              │                  │
                              └──────┬──────────┘
                                     ▼
                              rrf_fusion()  ← graph_search()
                                     │
                                     ▼
                              [record_search_metric]
                                     │
                                     ▼
                              RerankerService.rerank()
                                     │
                                     ▼
                              [record_rerank_metric]
                                     │
                                     ▼
                              CompressionService.compress()
                                     │
                                     ▼
                              [record_compress_metric]
                                     │
                                     ▼
                                   响应

                                      │
                                      ▼
                            QueryRouterService.route()
                                     │
                                     ▼
                              [record_route_metric]
```

### 4.2 模块划分

```
api/services/monitor/
├── __init__.py          # 导出外部接口
├── models.py            # MetricRecord 数据类 + 枚举
├── collector.py         # MetricCollector — 指标采集器（同步写入 kb_metrics）
├── aggregator.py        # MetricAggregator — 聚合查询（返回仪表盘 API 所需格式）
├── middleware.py         # FastAPI 中间件 — 自动采集每次请求的性能指标
└── router.py            # FastAPI router — 暴露监控 API 端点

api/routes/monitor.py    # 监控仪表盘路由入口
```

### 4.3 采集点分布

| 采集点 | 位置 | 触发方式 | 记录类型 |
|--------|------|----------|----------|
| `record_search` | `search()` 返回前 | 显式调用 | `search` |
| `record_rerank` | `rerank()` 完成后 | 显式调用 | `rerank` |
| `record_compress` | `compress()` 完成后 | 显式调用 | `compress` |
| `record_route` | `route()` 确定后 | 显式调用 | `route` |
| 请求耗时 | FastAPI middleware | 自动 | `request` |
| 错误 | `except` 块中 | 异常时触发 | `error` |

### 4.4 采集方式（同步 vs 异步）

| 方式 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| **同步写入** | 实现简单，数据实时 | 增加检索延迟（约 5-15ms） | ❌ 不选 |
| **异步写入** | 不阻塞主流程 | 需要队列/线程池 | ✅ **默认** |
| **日志转存** | 零性能影响 | 数据有延迟（分钟级） | ⬜ 备选 |

对于 MVP，使用 **`asyncio.create_task` + 同步数据库写入**（写入约 5-10ms，不阻塞检索响应）。

## 7. 关键设计决策

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 存储方式 | 独立 DB / 现有 PostgreSQL / 日志文件 | **现有 PostgreSQL（kb_metrics 表）** | 与 kb_chunks 同库，免运维，天然支持时序查询 |
| 采集方式 | 同步 / 异步 / 日志采集 | **异步（create_task）** | 同步增加检索延迟不可接受；日志采集延迟太大 |
| 采集粒度 | 每次查询 / 采样（1:10） / 仅错误 | **每次查询** | 监控系统需要完整数据才能做精准分析，数据量可控（~4000条/日） |
| 聚合时机 | 写入时预聚合 / 查询时实时聚合 | **查询时实时聚合** | 数据量小（百万级），实时聚合更灵活，无需额外维护聚合表 |
| 前端形式 | React 应用 / 静态 HTML / API-only | **API-only (MVP)** | 项目无前端框架，API + CLI 可满足运维需求；需要 UI 时再用 Chart.js 扩展 |
| 用户级权限 | 全量可见 / 仅看自己的查询 | **全量可见（Admin）** | 监控仪表盘是运维工具，默认管理员可见全部 |
| 降级策略 | 数据库不可用时丢弃指标 | **丢弃 + 日志警告** | 监控本身不能影响主流程 |

## 10. 已知局限与后续优化

| 局限 | 影响 | 改进方向 |
|------|------|----------|
| 精确度评估依赖隐式反馈 | 无前端点击事件的场景下 precision@k 不可用 | 引入 A/B 测试工具，人工标注评估集 |
| 仅支持 `metric_type` 维度检索 | 多维度交叉分析需额外索引 | 添加 `(metric_type, user_id, timestamp)` 组合索引 |
| 无自动过期清理 | 数据无限增长 | 创建定时任务（每月1日）删除 > 180 天的数据 |
| API-only，无 UI | 需要手动调 API 查看 | 用 Chart.js 生成静态仪表盘页面 |
| 无告警机制 | 异常需要人工发现 | 接入飞书/企业微信 Webhook 通知 |

### 11.2 与现有系统关系

```
kb_metrics  ←──  MetricsCollector.collect()  ←──  AdvancedRetrievalService
                    │                                    │
                    ▼                                    ▼
            kb_metrics 表                         检索核心逻辑
            (完全独立)                            (零侵入)
```

监控系统与被监控的检索系统**解耦**：
- `collect()` 方法通过 `try/except` 保护，确保任何异常不传播到检索流程
- 指标数据仅追加，不改写任何现有表
- 监控 API 路由注册在独立的 `monitor_router` 上
